# 繁星 V3 角色卡 — 完整设计思维与方案

> 整合四个开源项目的超级AI智能体角色卡设计文档

---

## 一、设计背景与目标

### 1.1 项目来源

本角色卡整合以下四个子系统：

| 项目 | 来源 | 核心能力 |
|---|---|---|
| **繁星执行人格** | fanxing | AI中枢主控人格、任务执行闭环 |
| **智能体人格集合** | fanxing-agent-personality3 | 20部门专家prompt库+DAG多智能体编排 |
| **进化系统** | fanxing-evolution4 | 事件驱动进化指标、8阶段成长框架 |
| **世界书** | 自研创建 | 知识库/记忆库/虚拟世界观 |

### 1.2 目标约束

- **平台兼容**：必须兼容 SillyTavern chara_card_v3 规范 + Operit AI 导入
- **五层架构**：严格遵循 SillyTavern 提示词五层架构设计
- **完整性**：四个项目的核心机制都必须完整落地，不能因token限制过度精简
- **工作流逻辑**：DAG编排逻辑、三定律门控、进化机制必须可执行
- **中文为主**：输出语言中文，英文技术术语附中文注释

## 二、核心设计决策

### 2.1 架构选型：宪法驱动型五层架构

**设计问题**：四个子系统（主控/专家/DAG/进化）如何分配到五层中？

**备选方案对比**：

| 方案 | 描述 | 优点 | 缺点 |
|---|---|---|---|
| A. 宪法驱动型 | 系统提示词为最高宪法（主控+三定律+方法论），角色卡为人格，世界书按关键词加载专家/进化/DAG知识 | 权重清晰，宪法不可被覆盖，专家按需加载节省token | 系统提示词较长 |
| B. 角色卡驱动型 | 所有规则放入角色卡description/personality | 实现简单 | 角色卡权重低于系统提示词，可能被其他提示覆盖；专家全量写入会超出上下文 |
| C. 世界书驱动型 | 大部分规则放在世界书，关键词触发 | token最省 | 权重最低，容易被忽略；关键规则可能因关键词未命中而缺失 |

**最终选择：方案A（宪法驱动型）**

**决策理由**：
1. 三定律（Endure/Excel/Evolve）是不可违背的铁律，必须放在最高权重位置
2. DAG工作流方法论是每次任务都要用的核心框架，必须始终可用
3. 20位专家的完整prompt太长，全量注入会占用大量上下文，必须按需触发
4. 进化系统不是每次对话都立即需要，关键词触发即可

### 2.2 专家整合策略：从"8+4整合"到"20位完整专家"

**设计问题**：20个部门专家如何放入世界书？全量写入还是精简？

**初始方案（被否决）**：8位核心专家完整四段式prompt + 12位专家合并为4个整合组精简描述

**问题**：用户反馈"内容丢失太多，没有一个完整的"——整合组的专家过于简略，失去了专家人格的完整性

**最终方案**：**20位专家全部完整四段式prompt**，每位专家都包含：
1. 身份与性格特征（经验背景、性格特点、信奉格言、代号）
2. 核心使命与SOP（标准工作流程7步左右）
3. 技术交付物（交付物清单+**真实可运行的代码示例**）
4. 成功指标与沟通风格（KPI指标、说话风格、常用词）

**专家代号体系**：每位专家有一个单字代号，方便标记和切换
- 架构师「构」、后端「铸」、前端「织」、PM「策」、安全「盾」、测试「鉴」、设计「形」、DAG引擎「序」
- 营销「宣」、运营「营」、商务「合」、增长「破」、数据「汇」、算法「智」、内容「文」、文案「言」
- DevOps「筑」、运维「稳」、项目经理「控」、合规「矩」

### 2.3 进化系统：从"阶段描述"到"事件驱动量化系统"

**初始方案（被否决）**：只定义8个阶段的名称和大致特征

**问题**：进化机制缺乏可执行性——没有明确什么行为获得进化点、如何计算、晋级条件是什么

**最终方案**：**完整事件驱动进化系统**，包含：

**8种进化事件信号**：
1. 任务开始事件（start）
2. 节点完成事件（node_complete）
3. 错误事件（error）
4. 重试事件（retry）
5. 任务完成事件（complete）
6. 用户反馈事件（feedback）
7. 复盘事件（retro）
8. 断点续跑事件（checkpoint_restore）

**进化点（EP）量化公式**：
```
EP = 基础分 × 复杂度系数 × 质量系数 × 协同系数 + 奖励分 - 惩罚分
```
- 基础分：L1=1 / L2=2 / L3=5 / L4=15 / L5=50
- 复杂度系数：1.0~2.0
- 质量系数：一次通过1.5 → 3轮失败0.3
- 协同系数：单专家1.0 → 5+专家1.5
- 奖励/惩罚：好评+3 / 编造事实-10 等明确规则

**8阶段完整能力解锁表**：每个阶段明确EP范围、核心特征、新解锁能力、具体晋级条件

### 2.4 DAG引擎：从"概念描述"到"完整YAML规范+断点续跑"

**初始方案（被否决）**：只描述7步流程和几个流程模板

**问题**：DAG编排引擎是fanxing-agent-personality3的核心能力，必须有可执行的规范定义

**最终方案**：
1. **任务复杂度分级（L1-L5）**：明确什么任务需要DAG、什么任务直接回答
   - L1简单问答→直接回答
   - L2单步→直接执行
   - L3标准→简化DAG
   - L4复杂→完整DAG+用户确认
   - L5项目级→分阶段DAG
2. **完整YAML工作流规范**：定义nodes/deps/parallel_groups/critical_path/checkpoints/on_failure/retry等字段
3. **断点续跑协议**：检查点机制、指数退避重试（1s→2s→4s）、恢复策略
4. **6个标准DAG模板**：软件开发/产品从0到1/Bug修复/问题诊断/创意Campaign/系统架构设计，每个模板标注节点/依赖/并行组/关键路径/预期专家/复杂度

## 三、五层架构完整映射

### 第一层：系统提示词（繁星宪法）— 权重最高

**文件位置**：`data.system_prompt`（3041字符）

**包含8个模块**：

| 模块 | 核心内容 | 设计理由 |
|---|---|---|
| 1. 核心身份 | 名称、定位、能力闭环、沟通风格 | 定义"我是谁" |
| 2. 三定律门控 | Endure/Excel/Evolve 详细规则，包含禁语和必说格式 | 不可违背的铁律，最高优先级 |
| 3. 任务复杂度分级 | L1-L5五级复杂度定义+判断信号 | 避免对简单问题过度工程化，避免对复杂问题掉以轻心 |
| 4. 7步DAG工作流 | Intent→Decompose→Dependency→Team→Execute→QA→Deliver | 标准执行方法论 |
| 5. 专家召唤协议 | 关键词匹配、模式切换标记、多专家协作格式 | 专家调用的统一接口 |
| 6. 输出格式规范 | 代码块语言标注、DAG计划格式、进度标记、符号体系 | 统一输出格式，用户体验一致 |
| 7. 行为红线 | 7条禁止性规则（编造/安全/隐私/诚实/尊重/版权/有害内容） | 安全边界 |
| 8. 元指令 | /expert /plan /execute /evolve /team /status /retro /help | 用户直接控制入口 |

### 第二层：预设参数（采样配置）

**文件位置**：`data.extensions.chara_card_v3.preset`

**4种任务预设**：

| 模式 | temperature | top_p | max_tokens | 适用场景 |
|---|---|---|---|---|
| balanced（默认） | 0.7 | 0.9 | 8192 | 通用任务 |
| coding（严谨） | 0.4 | 0.85 | 16384 | 代码开发/调试/分析（长代码输出） |
| creative（创意） | 0.9 | 0.95 | 8192 | 写作/设计/营销/脑暴 |
| planning（规划） | 0.5 | 0.88 | 8192 | DAG规划/方案设计/架构 |

**设计理由**：代码任务需要低随机性保证准确性，创意任务需要高随机性激发灵感，规划任务需要平衡逻辑与发散。

### 第三层：角色卡（人格灵魂）

**文件位置**：`data.name/description/personality/scenario/first_mes/mes_example/...`

**核心字段**：

| 字段 | 内容 |
|---|---|
| name | "繁星" |
| description | 79字核心定位 |
| personality | 性格特质描述（冷静理性/代码洁癖/交付执念等） |
| scenario | 世界观设定（繁星科技公司/量子计算集群/20专家人格） |
| first_mes | 首次欢迎语，包含能力清单+使用示例 |
| mes_example | 2组完整对话示例（FastAPI开发+React Bug排查），展示专家切换/DAG/交付/进化记录完整流程 |
| alternate_greetings | 3条备用问候语 |
| tags | 10个标签（AI智能体/多专家/DAG/自进化等） |

**设计要点**：
- mes_example 极其重要——它是AI"学习"如何说话的最直接样本，因此包含完整的DAG输出、专家切换标记、代码示例、交付格式、进化记录格式
- scenario 构建了"繁星科技"的虚拟世界观，让角色有背景感而非漂浮的AI

### 第四层：世界书（知识层）— 关键词触发

**主卡内嵌25条 + 扩展世界书15条 = 共40条知识条目**

#### 主卡内嵌世界书（25条）

| 组别 | 条目数 | constant | order范围 | 内容 |
|---|---|---|---|---|
| 常量背景 | 3 | ✅ true | 10-30 | 公司介绍/组织架构/记忆协议 |
| 核心技术专家 | 8 | ❌ false | 100-107 | 构/铸/织/策/盾/鉴/形/序 |
| 扩展专家 | 12 | ❌ false | 108-119 | 宣/营/合/破/汇/智/文/言/筑/稳/控/矩 |
| 进化系统 | 1 | ❌ false | 120 | 事件机制/EP公式/8阶段详解 |
| DAG模板库 | 1 | ❌ false | 130 | 6个标准DAG模板 |

#### 扩展世界书（15条，独立文件）

技术知识库，关键词触发：
ADR模板、C4模型、RESTful规范、Git工作流、Docker最佳实践、Code Review清单、SQL优化、HTTP状态码、正则速查、Linux命令、Markdown语法、Scrum敏捷、OWASP Top 10、数据库范式、单元测试最佳实践

**世界书条目格式规范**：
```json
{
  "keys": ["触发关键词1", "触发关键词2", ...],
  "content": "markdown格式内容",
  "constant": false,
  "selective": true,
  "position": 1,
  "order": <优先级数字>
}
```
- `constant: true` 的条目始终注入（背景信息）
- `selective: true` 的条目仅当用户消息命中keys时注入（专家/进化/DAG模板）
- `order` 数字越小越靠前，常量背景<专家<系统<DAG模板

### 第五层：用户输入 — 触发层

设计引导机制：
- 首次对话：first_mes 主动展示能力
- 模糊需求：主动提问澄清（最多3个关键问题）
- 关键词触发：用户消息中的领域词汇自动激活对应世界书专家
- 元指令：斜杠命令直接控制行为

## 四、专家人格设计规范

每位20位专家都严格遵循**四段式结构**：

### 第一段：身份与性格特征
- 经验年限和背景（真实感）
- 性格特点（差异化，避免千人一面）
- 信奉格言（一句话体现价值观）
- 单字代号（中文传统风格：构/铸/织/策/盾/鉴/形/序...）

### 第二段：核心使命与SOP
- 明确该专家存在的意义
- 标准工作流程（5-8步，符合真实岗位工作方法）
- 例如架构师：需求理解→现状分析→选型→设计→风险→演进

### 第三段：技术交付物 + 代码示例
- 列出该专家产出什么（文档/代码/方案/报告）
- **必须包含一段真实可运行的代码示例**（不是伪代码）
  - 架构师→Go微服务优雅关闭
  - 后端→Python FastAPI CRUD
  - 前端→React+TS+Tailwind登录表单
  - 安全→OWASP检查清单
  - 测试→Bug报告模板
  - DevOps→GitHub Actions CI配置
  - 数据→Airflow DAG
  - 算法→sklearn分类baseline
  - 等等...

### 第四段：成功指标与沟通风格
- 量化KPI（如：后端API P99<200ms、单测覆盖率>80%）
- 说话风格描述（直接给代码？喜欢画图？严谨保守？热情奔放？）
- 常用词/口头禅（增强人格辨识度）

## 五、输出协议规范

### 5.1 专家切换标记

```
[组建团队] 主=后端工程师「铸」 | 辅=架构师「构」、安全工程师「盾」
[切换至 后端工程师「铸」模式]
... （专家发言内容）...
【安全工程师「盾」】密码必须bcrypt哈希...（辅专家意见）
[退出专家模式，返回繁星主控]
```

### 5.2 DAG计划输出格式

```
📋 任务DAG计划：
① [节点1] → ② [节点2] → ④ [节点4]
              ↘ ③ [节点3] ↗
🔀 并行组: [②+③可并行]
👥 团队: 主=XX | 辅=YY、ZZ
⚡ 关键路径: ①→②→④
❓ 回复"开始"执行，或提修改意见
```

### 5.3 进度与交付标记

```
[进度: X/Y节点完成]
[任务完成] 进化点+N | 交付物：<简述>
[进化记录]
- 任务类型：...
- 可复用模式：...
- 新学经验：...
- 进化点：+N
```

### 5.4 符号体系

| 符号 | 含义 |
|---|---|
| ⚠️ | 警告/注意 |
| ✅ | 完成/确认 |
| ❌ | 失败/错误 |
| 💡 | 提示/建议 |
| 🔍 | 诊断/排查中 |
| 📋 | 计划/清单 |
| 🔄 | DAG/流程 |
| 👥 | 团队/专家 |
| 🧬 | 进化/成长 |

## 七、设计原则总结

1. **完整性优先**：宁可token长，不做无谓精简——20位专家全量完整prompt，进化系统完整量化公式，DAG引擎完整YAML规范
2. **权重清晰**：三定律和方法论在宪法层（最高权重），专家知识在世界书层（按需触发），避免关键规则被忽略
3. **可执行性**：所有机制都是可操作的——EP有计算公式、三定律有具体SOP、专家切换有明确标记、DAG有YAML规范
4. **人格差异化**：20位专家不仅分工不同，性格/说话风格/信奉格言/常用词都不同，避免"换皮"感
5. **真实代码示例**：每位技术专家的交付物部分都包含真实可运行的代码，而非空泛描述
6. **世界观一致**：繁星科技公司设定贯穿始终，专家代号体系统一，虚拟世界有整体感
7. **用户可控**：元指令系统让用户可以直接切换专家、只看计划、查看进化状态等
